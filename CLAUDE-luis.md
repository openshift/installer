## OpenShift Installer Overview

The OpenShift Installer provisions OpenShift clusters across multiple cloud platforms (AWS, Azure, GCP, OpenStack, vSphere, Bare Metal, etc.). It uses an asset dependency graph system and integrates with Terraform and Cluster API for infrastructure provisioning.

## Build Commands

Build the main installer binary:
```sh
hack/build.sh
```

Build with Cluster API bundled (optionally skip Terraform):
```sh
export SKIP_TERRAFORM=y  # optional
hack/build.sh
```

Build specific components:
```sh
hack/build-cluster-api.sh  # Build Cluster API components
hack/build-node-joiner.sh  # Build node-joiner binary
```

## Architecture Overview

### Asset Dependency Graph System

The installer uses a sophisticated asset dependency graph system (in `pkg/asset/`) where:
- Each asset (configuration, manifest, ignition config, etc.) declares its dependencies
- Assets are generated in dependency order
- The asset store (`pkg/asset/store/`) manages the lifecycle and caching

Key asset types:
- **InstallConfig**: User-provided cluster configuration
- **Manifests**: Kubernetes manifests for cluster components
- **Ignition Configs**: Machine configuration for CoreOS nodes
- **TLS**: Certificate assets for cluster components

### Infrastructure Provisioning Approaches

The installer supports two primary modes:

1. **IPI (Installer-Provisioned Infrastructure)**: Installer creates all infrastructure
   - Uses Terraform (`pkg/terraform/`) or Cluster API (`pkg/clusterapi/`)
   - Platform-specific implementations in `pkg/asset/cluster/`

2. **UPI (User-Provided Infrastructure)**: User creates infrastructure
   - Templates in `upi/` directory
   - Installer generates manifests and configs only

### Cluster API Integration

The installer uses Cluster API controllers via a **local control plane** powered by `kube-apiserver` and `etcd`:
- Local control plane setup via envtest (from controller-runtime)
- CRDs stored in `data/data/cluster-api/`
- Providers vendored to `cluster-api/providers/<provider-name>/`
- Built and packaged into `cluster-api.zip` during build process

See `docs/dev/cluster-api.md` for details.

### Platform-Specific Code Organization

Each platform has dedicated packages:
- **Types**: `pkg/types/<platform>/` - Platform configuration types
- **Validation**: `pkg/types/<platform>/validation/` - Input validation
- **Defaults**: `pkg/types/<platform>/defaults/` - Default values
- **Assets**: `pkg/asset/<component>/<platform>/` - Platform-specific asset generation
- **Terraform**: `data/data/<platform>/` - Terraform modules and configs
- **Cluster API Providers**: `cluster-api/providers/<platform>/` - CAPI provider implementations

### Command Structure

Main entry points in `cmd/openshift-install/`:
- `create.go` - Cluster creation commands (install-config, manifests, ignition-configs, cluster)
- `destroy.go` - Cluster destruction
- `main.go` - CLI setup with cobra

The `create cluster` workflow:
1. Generate install-config (or load existing)
2. Validate configuration
3. Generate manifests
4. Generate ignition configs
5. Provision infrastructure (via Terraform/CAPI)
6. Bootstrap cluster
7. Wait for cluster operators

## Dependency Management

Go dependencies are vendored using Go Modules:
```sh
go get <dependency>@<version>  # Add/update dependency
go mod tidy                    # Tidy go.mod and go.sum
go mod vendor                  # Re-vendor dependencies
```

**Important**: Commit vendored code separately from other changes.

### Updating CAPI Provider Dependencies

When bumping Cluster API provider versions:

1. Update provider `go.mod`:
```sh
cd cluster-api/providers/<provider>/
go get sigs.k8s.io/cluster-api-provider-<provider>/v2@<revision>
go mod tidy && go mod vendor
```

2. Update installer top-level `go.mod`:
```sh
cd <installer-root>
go get sigs.k8s.io/cluster-api-provider-<provider>/v2@<revision>
go mod tidy && go mod vendor
```

3. Update infrastructure CRDs in `data/data/cluster-api/<provider>-infrastructure-components.yaml`

If you touch `github.com/openshift/api`, also run:
```sh
go generate ./pkg/types/installconfig.go
```

## Commit Message Format

Follow this convention:
```
<subsystem>: <what changed>

<why this change was made>

Fixes #<issue-number>
```

- Subject line ≤70 characters
- Wrap body at 80 characters
- Examples: `aws: add support for new instance types`, `asset: fix dependency ordering bug`

## Import Order

Go imports are organized into 4 sections (standard, default, github.com/openshift prefix, blank):
```sh
hack/go-fmt.sh .  # Automatically sorts imports
```

## Key Directories

- `cmd/` - CLI entry points (openshift-install, node-joiner)
- `pkg/asset/` - Asset dependency graph implementation
- `pkg/types/` - Type definitions and validation
- `pkg/terraform/` - Terraform integration
- `pkg/clusterapi/` - Cluster API integration
- `pkg/destroy/` - Cluster destruction logic
- `cluster-api/providers/` - Cluster API provider implementations
- `data/data/` - Embedded assets (Terraform modules, manifests, CRDs)
- `upi/` - User-provided infrastructure templates
- `hack/` - Development scripts
- `docs/dev/` - Developer documentation

## Platform Support

Supported platforms: AWS, Azure, GCP, OpenStack, vSphere, Bare Metal, IBM Cloud, Power VS, Nutanix, z/VM

Each platform may support IPI, UPI, or both modes.
